WDF框架选型：从Storport到KMDF的移植之路

cursorhu
2026-02-21
Windows驱动开发
KMDF, SD卡, Storport, WDF, Windows驱动

概述

本文介绍将O2Micro SD Host Controller驱动从Storport框架移植到WDF（Windows Driver Frameworks）的选型决策，详细分析为什么选择KMDF作为基础框架，以及为什么不使用微软Inbox的SD WDF框架。

1. 项目背景

1.1 原架构：Storport驱动

原项目 bht-uefi-sdboot/storport/ 基于Windows Storport框架实现：

  • Storport.sys：Windows内核存储端口驱动
  • 优势：高性能、经过验证的存储驱动架构
  • 劣势：纯WDM架构，需要手动管理大量内核对象

1.2 移植目标：WDF框架

新项目 bht-uefi-sdboot/WDF-sdhcstor-shirley/ 采用WDF框架：

2. WDF框架选型分析

2.1 WDF框架简介

WDF（Windows Driver Frameworks）包含两个主要框架：

框架 全称 适用场景
KMDF Kernel-Mode Driver Framework PCI/USB/SD等设备驱动
UMDF User-Mode Driver Framework 用户态设备驱动

2.2 为什么选择KMDF

参考微软官方示例 PCIDRV，KMDF是PCI设备驱动的最佳选择：

PCIDRV示例展示的关键KMDF特性：

KMDF接口 PCIDRV应用 WDF-sdhcstor应用
PnP/Power事件 EvtDeviceAdd, EvtDeviceD0Entry ✅ 相同
硬件资源映射 EvtDevicePrepareHardware ✅ 相同
DMA接口 Scatter-Gather DMA ✅ ADMA2
中断处理 EvtInterruptIsr, EvtInterruptDpc ✅ SDHC中断
I/O队列 WdfIoQueueCreate ✅ SCSI请求队列

2.3 为什么不用Inbox SD WDF框架

查看 windows-hardware-drivers-sd.pdf，微软提供了Inbox SD框架：

Inbox框架的限制：

限制 说明 对本项目的影响
标准化协议栈 Sdbus.sys提供完整的SD协议栈 ❌ 无法实现私有SD Express流程
有限的定制能力 卡检测、初始化流程由Inbox控制 ❌ 需要软件控制的模式切换
不支持BBR方案 无法在驱动层实现热插拔模拟 ❌ 嵌入式平台需要BBR
调试困难 黑盒协议栈，问题定位困难 ❌ 需要完整的调试可见性

2.4 选型决策对比

特性 Inbox Sdbus.sys 自定义KMDF
协议栈控制 ❌ 不可控 ✅ 完全可控
SD Express支持 ⚠️ 有限 ✅ 可定制
BBR热插拔模拟 ❌ 不支持 ✅ 支持
调试可见性 ❌ 黑盒 ✅ 完全透明
开发复杂度 低 高
维护成本 低 高

最终选择：自定义KMDF框架

原因：项目需要实现私有的SD Host控制流程，包括：

  1. 软件控制的SD Express模式切换
  2. BBR（Bios Boot Record）热插拔模拟
  3. 完整的调试可见性

5. 与Storport的对比

5.1 Storport特点

特性 Storport KMDF
架构 WDM Pure WDF
性能 最高 高（接近Storport）
代码量 大 中等
学习曲线 陡峭 平缓
调试支持 一般 丰富（WDFVERIFY等）
PnP/Power 需手动实现 框架支持

5.2 移植收益

方面 改进
代码复用 复用原有SD协议栈代码
框架支持 WDF自动处理PnP/Power
调试工具 使用WDF内置调试宏
未来扩展 更容易添加新功能

6. 总结

本文分析了WDF框架选型的决策过程：

  1. 选择KMDF：基于PCIDRV示例，KMDF是PCI设备驱动的最佳选择
  2. 放弃Inbox SD：Inbox框架无法满足私有SD Express流程和BBR方案需求
  3. 架构设计：采用KMDF + 自定义SD协议栈的混合方案

这种选型决策平衡了：

  • 技术需求的满足度
  • 开发复杂度的可控性
  • 长期维护的成本
