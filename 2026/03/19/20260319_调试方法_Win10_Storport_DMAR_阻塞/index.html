分析更新 1：卡死时机分析 (2026-03-19)

1.1 原始结论 vs 新分析

分析角度 原始结论 当前分析
卡死时机 安装阶段（pnputil执行期间），驱动.sys尚未加载 可能不准确，需要Win10 setupapi.log验证
驱动代码 完全不执行 如果卡死在Starting device之后，驱动代码可能执行
根因位置 Storport框架处理DmaRemappingCompatible Storport框架或驱动DMA初始化代码

1.2 结论来源分析

原始分析基于以下假设：

假设依据：

  1. pnputil 是用户态程序
  2. 在 AddService 阶段，驱动 .sys 只是被注册为服务，尚未真正加载
  3. 因此驱动代码不会执行

问题：我们没有 Win10 实际卡死时的 setupapi.dev.log，这个结论是推断而非实证。

1.3 从 Win11 日志的观察

从 Win11 setupapi.log 的正常安装日志：

关键观察：

  1. Starting device 和 completed 之间几乎没有时间间隔（同一毫秒）
  2. Win11 上驱动加载/初始化是快速完成的
  3. 设备未启动原因是 CM_PROB_CONSOLE_LOCKED（控制台锁定），不是驱动错误

1.4 需要确认的关键问题

要确认 Win10 卡死点，必须：

  1. 在 Win10 上先开启 setupapi.dev.log
  2. 执行驱动安装
  3. 观察日志最后一条记录

可能的卡死场景：

场景 日志表现 说明
场景A 日志停在 Starting device 之前 PnP 框架卡死，驱动未加载
场景B 日志停在 Starting device 之后，很久才有结果 驱动代码卡死
场景C 日志显示 Starting device completed 但无后续 PnP后续流程卡死

2.2 潜在卡死点详细分析

卡死点 A：scsi_HwFindAdapter 中的 DMA 缓冲区分配（高优先级）

文件位置：SDSTORDriver/sdstor/windows_os/winscsientry.c:1186-1215

调用的底层函数 (SDSTORDriver/sdstor/windows_os/winapi.c:2004-2030)：

卡死原因分析：

  • 当 INF 声明了 DmaRemappingCompatible=1
  • StorPortGetUncachedExtension 会尝试分配符合 IOMMU/DMAR 要求的缓冲区
  • 在 Win10 上，这个过程可能触发同步等待（例如等待 IOMMU 响应）
  • 如果 IOMMU 无响应或超时机制不完善 → 整个调用卡死

卡死点 B：StorHwPassiveInitialize 中的 PowerFx 初始化（中优先级）

文件位置：SDSTORDriver/sdstor/windows_os/winscsientry.c:370-453

卡死原因分析：

  • StorPortInitializePoFxPower 是 Power Management Framework 初始化
  • 在 Win10 上可能与 DMA remapping 初始化有依赖关系
  • 如果 DMA remapping 初始化未完成，PowerFx 可能等待

卡死点 C：Storport 框架的 DMA Remapping 初始化（根本原因）

这是最可能的根因位置

2.3 为什么 Win11 不卡死？

方面 Win10 Win11
DMA Remapping 框架 较旧，可能有 bug 或不完整 更新，修复了多个问题
DmaRemappingCompatible 处理 可能同步等待 异步或修复了
IOMMU 集成 松散耦合 紧密集成
DMA 缓冲区分配 可能阻塞 非阻塞

2.4 建议的调试方向

步骤1：确认卡死场景（最优先）

在 Win10 上执行以下步骤：

  1. 先开启 setupapi.dev.log
  2. 执行驱动安装
  3. 检查日志最后记录

根据日志结果确定是场景A、B还是C。

步骤2：如果确认是驱动加载后卡死（场景B）

在 os_alloc_dma_buffer 入口添加调试打印：

验证卡在哪一次 StorPortGetUncachedExtension 调用。

步骤3：临时规避尝试

如果时间紧迫，可以尝试：

  1. 禁用 DMA 缓冲区分配：

    • 在 scsi_HwFindAdapter 中注释掉 os_alloc_dma_buffer 调用
    • 如果不分配 DMA 缓冲区就不卡死 → 确认是 DMA 相关问题

  2. 临时移除 DmaRemappingCompatible：

    • 修改 INF 将值设为 0
    • 验证卡死是否消失

3.1 驱动加载相关函数

函数名 文件:行号 功能 卡死风险
DriverEntry winscsientry.c:2675 驱动入口点 低
scsi_HwFindAdapter winscsientry.c:989 适配器查找和初始化 高
os_alloc_dma_buffer winapi.c:2004 DMA缓冲区分配 最高
StorHwPassiveInitialize winscsientry.c:370 被动初始化回调 中
scsi_HwInitialize winscsientry.c:488 硬件初始化 低

3.2 DMA 相关 API

API 说明 与DMAR的关系
StorPortGetUncachedExtension 分配公共缓冲区 关键：在DMAR环境下分配符合要求的缓冲区
StorPortAllocateDmaMemory 分配DMA内存 返回IOVA而非PA
StorPortMapDmaTransfer DMA映射传输 使用IOVA
StorPortGetD3ColdSupport 查询D3Cold支持 可能与DMAR交互
StorPortInitializePoFxPower PowerFx初始化 可能与DMAR有依赖

常见问题排查清单

安装阶段卡死

  • DIFx 日志是否记录到最后一步
  • Process Monitor 显示最后访问的注册表键
  • 断点是否命中

内核调试无响应

  • 串口连接是否正常
  • 波特率是否匹配
  • 目标机是否正确进入调试模式

符号加载失败

  • 网络连接是否正常
  • 尝试使用本地缓存
  • 使用 .symfix 和 .reload

ETW 跟踪为空

  • 检查 Provider GUID 是否正确
  • 使用 logman query -ets 查看活动会话
  • 确认管理员权限
